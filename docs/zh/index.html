Spend Request → Policy → Decision → Ledger

这到底是什么？

Jarvis Worker 不是一个“替你下单”的花钱机器人。更准确地说，它是一套给 Agent 用的 钱包防火墙 + 财务官 + 审计账本：先写清规则，再允许花钱；花完以后必须留下证据，能被复盘，也能被追责。

一句话定位：
让 Agent 在规则约束下获得消费能力， 从“只会产出”的工具，变成一个能被制度化管理的经济角色。
1. 先定义支出目标
这笔钱为什么要花？服务于哪个交付？预期提升什么？
2. 再经过规则判断
预算上限、品类白名单、风险等级、人工审批阈值、异常冻结。
3. 最后写进账本
每笔支出都留下原因、对象、证据和后续复盘入口。

为什么先从“花钱”讲起？

大多数 AI 叙事都停在供给侧：提效、降本、替代岗位。可一旦你只谈“产出”， Agent 依然更像工具。只有当它开始稳定、可控、可追责地支出，经济闭环才算真正跑起来。

  • 只会赚钱的 Agent，更像工具；它主要改变的是供给侧效率。
  • 会花钱的 Agent，才有机会成为需求侧变量，把钱重新打回工具、服务、质检、人类时间与公益等环节。
  • 但前提是：这不是“自由花钱”，而是被规则锁住的消费能力。

它不是“自动付款”，而是“制度化消费”

这套系统真正想解决的，是“钱怎么在边界内流动起来”。因此，产品核心不在支付按钮，而在三个层面：

  • 预算闸：总预算、单笔限额、预算桶、风险分级。
  • 权限闸：商家白名单、类目白名单、关键动作 human-in-the-loop。
  • 责任闸：每笔支出都要绑定原因、交付、预期提升点，并写入可审计账本。
我们想展示的，不是“它买了什么”，而是“它如何在规则内花钱”。

这套系统最适合先落在哪些场景？

  • 可靠性消费：为抽检、复核、复现、审计和人类 review 买单。
  • 人类服务支出：在转型期购买人类时间，而不是只购买更多算力。
  • 小额公益/定向预算：让支出带上用途标签、回执要求与异常冻结规则。
  • 更远一点：做成带用途标签的预算执行器，解决“钱明明在账上，但服务没有发生”的问题。
